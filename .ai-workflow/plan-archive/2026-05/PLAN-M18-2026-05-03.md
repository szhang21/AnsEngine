# Plan Archive: PLAN-M18-2026-05-03

## GoalSummary

- 本轮目标：在 M17 scripting foundation 之上加入最小真实输入能力，让脚本可以读取按键状态，并通过内置 `MoveOnInput` 脚本驱动自身对象在世界 XZ 平面移动。
- 成功标准（可验证）：
  - `Engine.Platform.InputSnapshot` 能表达 `W/A/S/D` 按键状态。
  - `Engine.Scripting.ScriptContext` 每帧携带 scripting 自有 input snapshot。
  - `ScriptRuntime.Update(...)` 将同一帧 input 传给所有 bound scripts。
  - 内置 `MoveOnInput` 能根据 `W/A/S/D` 修改自身 `Transform.Position`。
  - 多键斜向移动会归一化，不产生更快斜向速度。
  - `MoveOnInput` 修改后的 Transform 可由 runtime snapshot 与 render frame 观察。
  - `Engine.Scripting` 不依赖 `Engine.Platform`。
  - `Engine.Scene` 不依赖 `Engine.Scripting`。
  - `Engine.Render` 不感知 input、script 或 `MoveOnInput`。

## PlanningDecisions

- M18 目标是 Scripting Runtime Interaction MVP，不是 Editor bridge。
- 第一版按键状态只支持 `W/A/S/D`。
- 移动空间固定为世界 XZ 平面。
- `W` 对应 Z 负方向。
- `S` 对应 Z 正方向。
- `A` 对应 X 负方向。
- `D` 对应 X 正方向。
- `MoveOnInput` 的唯一必需属性是 `speedUnitsPerSecond: number`。
- 多键合成方向，非零方向归一化。
- `MoveOnInput` 只修改自身 Transform position，保持 rotation/scale 不变。
- `Engine.Scripting` 不直接依赖 `Engine.Platform`。
- `Engine.App` 是唯一 Platform input 到 Scripting input 的转换层。
- `Engine.Scene` 不新增 scripting dependency。
- Scene JSON `Script` component schema 沿用 M17，不新增字段。
- Editor 在 M18 只需要继续 preserve Script component。

## PlatformInputDesign

- Add `EngineKey` in `Engine.Platform`.
- M18 key set:
  - `W`
  - `A`
  - `S`
  - `D`
- Extend `InputSnapshot`:
  - keep `AnyInputDetected` public behavior.
  - add `IsKeyDown(EngineKey key)`.
  - support empty input, single key, multiple keys.
- Preferred implementation:
  - keep `InputSnapshot` a small readonly value object.
  - represent pressed keys with a small internal bit mask or immutable key set.
  - derive `AnyInputDetected` from pressed keys, or enforce consistency in constructors.
  - avoid exposing mutable collections.
- `NullInputService` returns empty input.
- If native OpenTK key polling is risky, M18 may land contract + stub/test input first; real native polling can be deferred, but OpenTK enum must not leak into `Engine.App` / `Engine.Scripting`.

## ScriptingInputDesign

- Add scripting-owned input types:
  - `ScriptKey`
  - `ScriptInputSnapshot`
- M18 `ScriptKey` set:
  - `W`
  - `A`
  - `S`
  - `D`
- `ScriptInputSnapshot` should expose:
  - `AnyInputDetected`
  - `IsKeyDown(ScriptKey key)`
  - empty input factory/default.
- `Engine.Scripting` must not reference `Engine.Platform`.
- `ScriptContext` gains `Input`.
- Binding-time context fields remain:
  - `ObjectId`
  - `ObjectName`
  - `SelfTransform`
  - `Properties`
- Frame-time context fields become:
  - `DeltaSeconds`
  - `TotalSeconds`
  - `Input`
- Replace or extend `ScriptContext.WithTiming(...)` with a frame update method, for example `WithFrame(double deltaSeconds, double totalSeconds, ScriptInputSnapshot input)`.
- `ScriptRuntime.Update(...)` should accept input:
  - recommended shape: `Update(double deltaSeconds, double totalSeconds, ScriptInputSnapshot input)`.
  - update all bound scripts in existing binding order.
  - pass the same input snapshot to every script in the same frame.
  - keep M17 fail-fast behavior for script exceptions.

## PropertyHelperDesign

- Add a small property reader in `Engine.Scripting` so built-in scripts do not hand-roll dictionary and type checks.
- Required helpers:
  - read required number.
  - read required boolean.
  - read required string.
- Optional helpers are out of scope unless implementation needs them for cleanup.
- Failure semantics:
  - missing property fails with a diagnostic naming script id, object id, and property name.
  - type mismatch fails with expected type and actual type.
  - non-finite numbers fail.
- Implementation guidance:
  - prefer explicit result/failure over ad hoc exception strings when binding can catch the failure early.
  - if an internal script throws for invalid property, `ScriptRuntime` must still translate it to stable `ScriptFailure`.
  - property validation for required built-in script fields should occur during `Initialize` / bind path, before the first render.
  - migrate `RotateSelf` to use the same helper for `speedRadiansPerSecond`.
  - `MoveOnInput` uses the helper for `speedUnitsPerSecond`.

## AppIntegrationDesign

- `Engine.App` remains the composition boundary between Platform and Scripting.
- Add an input conversion helper in App:
  - `InputSnapshot` -> `ScriptInputSnapshot`.
  - map `EngineKey.W/A/S/D` to `ScriptKey.W/A/S/D`.
- Main loop ordering remains:
  1. `ProcessEvents`
  2. `IInputService.GetSnapshot`
  3. `ITimeService.Current`
  4. `SceneRuntime.Update(time, input)`
  5. `ScriptRuntime.Update(time.DeltaSeconds, time.TotalSeconds, scriptInput)`
  6. `RenderFrame`
  7. `Present`
- Do not pass `ScriptInputSnapshot` to Scene.
- Do not pass `InputSnapshot` to Scripting.
- Register built-in scripts:
  - `RotateSelf`
  - `MoveOnInput`
- App failure behavior remains:
  - unknown script id fails before render.
  - invalid property fails before render.
  - script update exception fails before that frame's render.
  - renderer shutdown and window dispose still run.

## MoveOnInputDesign

- Built-in script id: `MoveOnInput`.
- Required property:
  - `speedUnitsPerSecond: number`
- Runtime behavior:
  - read `context.Input`.
  - compute direction:
    - `W`: `(0, 0, -1)`
    - `S`: `(0, 0, 1)`
    - `A`: `(-1, 0, 0)`
    - `D`: `(1, 0, 0)`
  - combine pressed directions.
  - if direction length is zero, do nothing.
  - normalize non-zero direction.
  - apply `position += direction * speedUnitsPerSecond * deltaSeconds`.
  - preserve rotation and scale.
- Strict non-goals:
  - no acceleration.
  - no damping.
  - no collision.
  - no gravity.
  - no camera-relative movement.
  - no object lookup.
  - no render/material/mesh access.

## Risks

- Input contract expansion can leak OpenTK details upward if not isolated.
- `AnyInputDetected` can drift from pressed key state if represented as an independent mutable flag.
- Required property validation can become inconsistent if each built-in script hand-rolls checks.
- `MoveOnInput` can accidentally introduce gameplay semantics like acceleration/collision if scope is not held tight.
- Native key polling may be fragile in headless CI; M18 should not require native window input to prove the runtime scripting path.

## Assumptions

- M18 only supports `W/A/S/D` key state.
- M18 movement is world-space XZ movement.
- M18 does not alter M17 Script component schema.
- M18 does not require Editor to add or edit Script components.
- M18 may use test/stub input to validate runtime behavior if native input polling is not yet stable.
- M18 starts after M17 is fully closed.
