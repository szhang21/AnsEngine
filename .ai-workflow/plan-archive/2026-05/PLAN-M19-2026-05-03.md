## TechnicalDesign

### SceneData Schema

- `RigidBody` component:
  - `type`: `"RigidBody"`
  - `bodyType`: `"Static"` or `"Dynamic"`
  - `mass`: optional number
- `BoxCollider` component:
  - `type`: `"BoxCollider"`
  - `size`: `{ "x": number, "y": number, "z": number }`
  - `center`: optional `{ "x": number, "y": number, "z": number }`, default zero
- Validation:
  - unknown `bodyType` fails.
  - `size` missing or any axis <= 0 fails.
  - non-finite `size` / `center` / `mass` fails.
  - Dynamic mass, if present, must be > 0.
  - SceneData validates schema and numeric semantics only; it does not run physics and does not check mesh/material existence.

### Engine.Physics Public Shape

- Suggested public types:
  - `PhysicsWorld`
  - `PhysicsWorldDefinition`
  - `PhysicsBodyDefinition`
  - `PhysicsBoxColliderDefinition`
  - `PhysicsTransform`
  - `PhysicsStepContext`
  - `PhysicsWorldSnapshot`
  - `PhysicsBodySnapshot`
  - `PhysicsBodyType`
  - `PhysicsAabb`
  - `PhysicsQueryResult` or equivalent
- Suggested world behavior:
  - load from physics-owned `PhysicsWorldDefinition`.
  - store body id/name, body type, initial pose, collider center/size and computed AABB.
  - expose `BodyCount`, step count, accumulated fixed seconds.
  - expose read-only snapshots and query results.
- M19 should prefer explicit result types for malformed physics input. If implementation uses exceptions, tests must pin exact failure kind/message enough for diagnostics.

### Data Flow

1. Scene JSON is loaded by `Engine.SceneData`.
2. SceneData validates and normalizes `Transform`, `RigidBody`, `BoxCollider`.
3. Test-only adapter maps a realistic SceneData fixture to `PhysicsWorldDefinition`.
4. `Engine.Physics` consumes only `PhysicsWorldDefinition`.
5. `PhysicsWorld` builds internal body/collider state.
6. Tests call fixed `Step(...)`.
7. Tests inspect `PhysicsWorldSnapshot` and query APIs.

### AABB Rules

- M19 uses Box/AABB only.
- Rotation is ignored for AABB calculation in M19.
- AABB center = Transform position + collider center.
- AABB half extents = component-wise positive box size * absolute transform scale * 0.5.
- Non-positive transform scale should fail or normalize only if existing SceneData already guarantees positive scale. The task card must pin the chosen behavior from current SceneData validation.
- Ground query uses a fixed ground plane at Y = 0 for query purposes only. It does not clamp positions or modify world state.

### Design Patterns

- Module boundary pattern: Physics owns physics world internals; SceneData owns file/schema; App will later compose.
- Adapter pattern: M19 may use a test-only adapter from SceneData descriptions to Physics definitions; production bridge belongs to M20.
- Snapshot pattern: external tests and future App/Scene integration observe read-only physics state.
- Query object/result pattern: overlap and ground checks return explicit result values instead of exposing internal body collections.
- Foundation/MVP split: M19 builds physics world and queries; M20 wires App/Scene runtime behavior and Transform writeback.

### Boundary Rules

- Allowed dependencies for `Engine.Physics`:
  - `.NET` standard library
  - `System.Numerics`
- Forbidden dependencies for `Engine.Physics`:
  - `Engine.SceneData`
  - `Engine.Contracts`
  - `Engine.Core`
  - `Engine.Scene`
  - `Engine.App`
  - `Engine.Render`
  - `Engine.Scripting`
  - `Engine.Editor`
  - `Engine.Editor.App`
  - OpenTK / ImGui / OpenGL
- Forbidden reverse awareness:
  - `Engine.Scene` must not reference `Engine.Physics` in M19.
  - `Engine.SceneData` must not reference `Engine.Physics`.
  - `Engine.Render` must not mention Physics.
  - `Engine.Scripting` must not mention Physics.

## Risks

- High: foundation may accidentally become Runtime MVP by writing Scene Transform or App loop integration.
  - Mitigation: mark Transform writeback and App scheduling as explicit M20 scope.
- Medium: Physics schema may overfit future solver needs too early.
  - Mitigation: keep `RigidBody` and `BoxCollider` minimal.
- Medium: AABB semantics with scale/rotation can become ambiguous.
  - Mitigation: M19 ignores rotation and pins scale behavior in tests.
- Low: SceneData and Physics duplicate type concepts.
  - Mitigation: accept duplication at the boundary; Physics owns runtime-neutral definitions, SceneData owns file/schema descriptions, and M20 bridge maps between them.
