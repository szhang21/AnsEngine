## GoalSummary

- 本轮目标：把当前 Editor 从调试面板升级为 Unity-like 场景编辑工具，补齐 `Script`、`RigidBody`、`BoxCollider` authoring，并把中央 Workspace 替换为最小 Scene View 预览。
- 成功标准：
  - 在 Editor 中创建或编辑一个 `MoveOnInput + Dynamic RigidBody + BoxCollider` mover 和 `Static RigidBody + BoxCollider` wall。
  - 保存后运行 App，mover 被 wall 阻挡，证明编辑出的 scene 能进入真实 runtime physics 主路径。
  - Editor 中央区域能看到当前 scene 的非空预览，不再只是占位文字。
- M21 不做 Gizmo、鼠标 picking、Undo/Redo、Project Browser、Play Mode、trigger gameplay 或 dynamic gravity/solver。

## Context

- M12 完成 `Engine.Editor` headless core，提供 `SceneEditorSession`、dirty、selection、保存与 reload 语义。
- M13 完成最小 GUI Editor shell，包含 Toolbar、Hierarchy、Inspector、Status Bar 和基础文件/对象工作流。
- M16 完成 component-oriented SceneData / Scene / Editor bridge，Editor Inspector 已迁移到组件组，但目前主要覆盖 `Object`、`Transform`、`MeshRenderer`。
- M17/M18 完成 scripting foundation 与 `MoveOnInput` 输入脚本主路径。
- M19/M20 完成 Physics foundation 与 runtime collision MVP，`RigidBody + BoxCollider` 已能进入 App physics bridge 并阻挡 kinematic movement。
- 当前痛点：Editor 还不能 author runtime 已支持的 Script/Physics 组件，也没有真实 viewport，视觉仍像调试工具而不是专业编辑器。

## TechnicalDesign

### Dependency Direction

- `Engine.Editor`:
  - allowed: current dependencies on `Engine.Core`, `Engine.Contracts`, `Engine.SceneData`
  - forbidden: `Engine.Editor.App`, `Engine.Render`, `Engine.Asset`, `Engine.Scene`, `Engine.App`, OpenTK, ImGui, OpenGL
- `Engine.Editor.App`:
  - allowed after M21 boundary update: `Engine.Editor`, `Engine.SceneData`, `Engine.Contracts`, `Engine.Platform`, `Engine.Scene`, `Engine.Render`, `Engine.Asset`
  - owns GUI composition, preview composition and editor-window lifetime
  - must not copy SceneData JSON editing business rules
- `Engine.Render`:
  - must not reference Editor or Editor.App
  - continues to consume `Engine.Contracts.SceneRenderFrame` / `ISceneRenderContractProvider`
- `Engine.App`:
  - runtime path must not depend on Editor or Editor.App

### Session Component Editing API

- Reuse existing file-model component definitions:
  - `SceneFileScriptComponentDefinition`
  - `SceneFileRigidBodyComponentDefinition`
  - `SceneFileBoxColliderComponentDefinition`
- Add `SceneEditorSession` methods:
  - `UpdateObjectScriptComponent(string objectId, SceneFileScriptComponentDefinition script)`
  - `RemoveObjectScriptComponent(string objectId, string scriptId)`
  - `UpdateObjectRigidBodyComponent(string objectId, SceneFileRigidBodyComponentDefinition rigidBody)`
  - `RemoveObjectRigidBodyComponent(string objectId)`
  - `UpdateObjectBoxColliderComponent(string objectId, SceneFileBoxColliderComponentDefinition boxCollider)`
  - `RemoveObjectBoxColliderComponent(string objectId)`
- Implementation should follow existing `UpdateObjectTransformComponent` / `UpdateObjectMeshRendererComponent` pattern:
  - validate document exists
  - update component list through `UpdateObjectComponents(...)`
  - normalize document
  - update `mScene`
  - set `IsDirty`
  - preserve selection where possible
- `RigidBody` and `BoxCollider` may exist independently and save successfully; this does not imply the object enters `PhysicsWorld`.

### Editor.App Controller And Inspector

- Extend `EditorAppController` with wrapper methods for the new session APIs and continue using `CaptureResult(...)` for `LastError`.
- Extend `EditorInspectorSnapshot` into Unity-like component groups:
  - `Object`
  - `Transform`
  - `MeshRenderer`
  - `Scripts`
  - `RigidBody`
  - `BoxCollider`
  - `PhysicsParticipation`
- `PhysicsParticipation` should report whether the selected object has all required runtime physics inputs:
  - `Transform`
  - `RigidBody`
  - `BoxCollider`
- Script authoring v1:
  - show script list for the selected object
  - at minimum edit the first script component's `scriptId`
  - edit script properties as a JSON object text area
  - parse to `IReadOnlyDictionary<string, SceneFileScriptPropertyValue>` on Apply
  - parse failures update `LastError` and leave the document unchanged
- RigidBody authoring v1:
  - body type combo: `Static` / `Dynamic`
  - mass numeric input
  - default `Dynamic` mass: `1`
  - default `Static` mass: `0` or omitted according to existing SceneData normalization behavior
- BoxCollider authoring v1:
  - size `InputFloat3`
  - center `InputFloat3`
  - default size: `(1, 1, 1)`
  - default center: `(0, 0, 0)`
- Component add/remove buttons should live near each component group title.
- Do not remove `Transform` through the v1 Inspector; object positioning remains a foundational authoring assumption.

### Unity-like Shell And Theme

- Upgrade layout to a stable Unity-like shell:
  - top Toolbar: compact, roughly 48-56px
  - left Hierarchy: about 22% width
  - center Scene View: primary workspace
  - right Inspector: about 32% width
  - bottom Status / Console-lite: roughly 34-48px
- Extend or rename `EditorGuiLayoutSnapshot.MainWorkspace*` concepts to `SceneView*` in a testable way.
- Apply an ImGui dark tool theme in `EditorGuiRenderer` or a small helper:
  - professional dark panel colors
  - clear selected row highlight
  - low or modest corner radius
  - consistent `FramePadding`, `ItemSpacing`, title spacing and panel borders
- Remove the current placeholder text: `Viewport is intentionally reserved for a later milestone.`

### Scene View Preview

- Add `EditorScenePreviewHost` or equivalent internal helper in `Engine.Editor.App`.
- Preview host responsibilities:
  - consume `SceneEditorSession.Scene`
  - own a preview `SceneGraphService` or narrow adapter
  - load the normalized scene into preview scene state
  - use `DiskMeshAssetProvider` or an equivalent sample asset provider for preview meshes
  - render a nonblank edit-time scene preview
- Refresh preview when:
  - a scene is opened
  - Apply succeeds
  - Save/SaveAs succeeds
  - selection changes, if selection visualization is UI-level or preview-level
- M21 Scene View does not execute scripts, run physics, call `ApplicationHost`, or enter Play Mode.
- Preferred render strategy:
  - reuse `Engine.Render` draw path through existing `SceneRenderFrame` contracts where feasible
  - if embedding into an ImGui child region needs viewport/scissor plumbing, keep the embedding code in `Engine.Editor.App`
  - avoid changing `Engine.Render` public contracts unless absolutely required
- Selection visualization:
  - required: Hierarchy/Inspector selected state remains clear
  - optional: Scene View visual highlight only if it does not require widening render contracts in M21

## Risks

- Scene View embedding may require OpenGL viewport/scissor isolation from ImGui rendering.
  - Mitigation: keep preview code inside `Engine.Editor.App`; use staged delivery if needed, but M21 cannot close without visible nonblank preview.
- Script properties as JSON text is less polished than a typed property grid.
  - Mitigation: accept JSON text for M21 to avoid inventing a property schema UI too early.
- Expanding `Engine.Editor.App` dependencies can blur module boundaries.
  - Mitigation: update boundary contract explicitly and add boundary tests proving `Engine.Editor` remains headless.
- Previous M20 metadata inconsistency has been reconciled before M21 execution.
  - Mitigation: if future workflow metadata drifts again, stop at dispatch boundary and reconcile via Steward/Human before execution.

## Assumptions

- Visual direction is Unity-like, not Godot-like.
- Scene View is edit-time preview only.
- M21 continues the no dynamic physics decision: no gravity, velocity, force, impulse or solver.
- `Engine.Editor.App` may become GUI + preview composition root.
- `Engine.Editor` remains business/session core only.
