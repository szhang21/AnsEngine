  - 状态：Planned

  - 优先级：P1

  - 推荐 API：
    - `PhysicsKinematicMoveResult ResolveKinematicMove(string bodyId, PhysicsTransform desiredTransform)` 或等价显式 result。

## TechnicalDesign

### Dependency Direction

- `Engine.Physics`:
  - allowed: .NET standard library, `System.Numerics`
  - forbidden: any Engine module, OpenTK, ImGui, OpenGL
- `Engine.Scene`:
  - allowed: current dependencies on `Engine.Core`, `Engine.Contracts`, `Engine.SceneData`
  - forbidden: `Engine.Physics`
- `Engine.App`:
  - owns composition and may reference `Engine.Physics`
  - owns production bridge and runtime order wiring

### Scene Writeback Contract

- `Engine.Scene` should expose a public writeback operation by `objectId`.
- The API must be generic runtime transform functionality, not named after Physics.
- It must use `Engine.Contracts.SceneTransform` as external value type.
- Invalid transform should fail explicitly:
  - non-finite position / scale / rotation
  - invalid quaternion if existing project conventions already validate it
- Missing object and missing Transform are not success.

### App Production Bridge

- App bridge converts `SceneDescription` to `PhysicsWorldDefinition`.
- It reads:
  - `SceneObjectDescription.ObjectId`
  - `SceneObjectDescription.ObjectName`
  - `SceneObjectDescription.LocalTransform`
  - `SceneRigidBodyComponentDescription`
  - `SceneBoxColliderComponentDescription`
- It maps:
  - SceneData `Static` -> Physics `Static`
  - SceneData `Dynamic` -> Physics `Dynamic`, but M20 treats it as kinematic if driven by Scene/Script
- The bridge must not mutate `SceneDescription`.
- Production bridge failures should be deterministic and testable.

### Kinematic Collision Model

- M20 treats script/logic movement as proposed movement.
- Physics receives latest desired transform and returns resolved transform.
- Static bodies are environment blockers.
- Dynamic bodies are not automatically moved by Physics in M20.
- Body transform ownership:
  - Scene/Script owns desired transform.
  - Physics owns collision resolution result.
  - App writes resolved transform back to Scene.
- Snapshot remains read-only.

### Runtime Data Flow

1. App loads scene JSON through `Engine.SceneData`.
2. App initializes `Engine.Scene`.
3. App builds `PhysicsWorld` through production bridge.
4. App binds scripts.
5. Each frame, scripts update Scene transforms first.
6. App reads candidate kinematic transforms from Scene runtime.
7. App calls Physics resolve/query.
8. App writes resolved transforms back to Scene.
9. Render frame is built from Scene after writeback.

### Failure And Fallback

- If physics bridge fails during initialization, App run should fail with diagnostic output.
- If a runtime object disappears or lacks Transform during writeback, bridge should fail visibly in tests; M20 does not need dynamic object creation/destruction support.
- If collision resolution is ambiguous, task cards must choose a conservative axis strategy before execution.
- If implementation requires `Engine.Scene -> Engine.Physics`, stop and raise boundary change request.

## PlanningDecisions

- M20 chooses Static Collision + Kinematic Movement MVP, not Dynamic Gravity MVP.
- App is the composition root for Physics runtime integration.
- Scene writeback is intentionally generic so future Animation/Play Mode can reuse it.
- Physics remains independent and does not know SceneData or Scene runtime objects.
- `MoveOnInput` is the preferred real runtime smoke path because it already proves input -> script -> transform.
- Dynamic body simulation, solver and visible gravity are deferred to M21+.

## Risks

- Risk: Kinematic resolve strategy becomes underspecified.
  - Mitigation: task cards must pin exact axis/minimum-translation behavior before Executor starts.
- Risk: Script and Physics fight over Transform ownership.
  - Mitigation: fixed order is Script -> Physics -> Render; Physics is final per-frame constraint.
- Risk: App bridge grows too large inside `ApplicationBootstrap.cs`.
  - Mitigation: use small internal App classes for production bridge and runtime adapter.
- Risk: M20 accidentally expands into dynamic physics.
  - Mitigation: Dynamic bodies are kinematic candidates only; no velocity/force/gravity tasks.

## TestPlan

- Scene tests:
  - transform writeback succeeds by object id.
  - writeback affects render frame and runtime snapshot.
  - missing object / missing Transform fail explicitly.
  - Scene boundary still forbids Physics references.
- Physics tests:
  - kinematic move with no collision returns desired transform.
  - kinematic move into static box returns resolved transform outside collider.
  - static bodies do not move.
  - invalid body id / invalid transform fail fast.
  - boundary confirms zero Engine references.
- App tests:
  - production bridge maps real `SceneDescription` to `PhysicsWorldDefinition`.
  - App update order is Script -> Physics -> Render.
  - `MoveOnInput` cannot move through static collider.
  - scene without physics components preserves existing movement behavior.
- Smoke:
  - headless app run with sample scene exits successfully and captures render Transform after physics writeback.
