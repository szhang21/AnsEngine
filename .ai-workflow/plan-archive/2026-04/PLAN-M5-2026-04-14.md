# Plan Archive: PLAN-M5-2026-04-14

## GoalSummary

- 本轮目标：在既有 `Scene -> Contracts -> Render` 解耦链路上落地“可变换渲染”，让渲染输出具备位置/缩放（可选旋转）能力。
- 成功标准（可验证）：
  - 契约层新增变换字段且保持向后兼容。
  - Scene 能输出包含变换信息的渲染帧。
  - Render 正确消费变换并输出可观察结果。
  - 依赖方向保持 `Engine.Render -> Engine.Contracts`，不回退到 `Engine.Render -> Engine.Scene`。

## Milestones

- M5: 可变换渲染能力扩展（唯一里程碑）
  - 交付结果：
    - 在 `Engine.Contracts` 渲染模型中增量加入 transform 字段（最小集：Position、Scale）。
    - `Engine.Scene` 提供变换化的渲染输入数据。
    - `Engine.Render` 在提交路径中应用变换计算，并保持 identity 默认行为。
    - `Engine.App` 继续只做装配与编排，不承载变换计算。
  - 验收口径：
    - Build: `dotnet build -c Debug` / `dotnet build -c Release` 通过。
    - Test: `dotnet test` 通过；新增契约兼容与变换提交测试通过。
    - Smoke: 运行可见变换效果且可稳定退出。
    - Perf: 相比 M4b 无明显帧时间退化。

## PriorityOrder

- P0: M5（契约兼容扩展 + 变换渲染落地）
- P1: 变换回归与依赖门禁验证
- P2: 暂不启用
- P3: 暂不启用

## Risks

- high: 契约字段扩展不兼容导致旧路径回归。
  - 规避策略：采用增量字段 + identity 默认值，保留现有字段语义。
- medium: 变换计算落在错误层级导致职责污染。
  - 规避策略：Scene 产数据、Render 做消费，App 仅编排。
- medium: 渲染结果正确但性能抖动。
  - 规避策略：增加最小性能对比验证并限制每帧额外分配。

## PlanningDecisions

- 默认策略：契约做“兼容扩展”，不做一次性重构。
- 范围边界：本轮不扩展材质系统与资源管线，仅聚焦 transform 能力。
- 质量策略：每一步保持可编译、可测试、可运行，避免大爆改。

## HandoffToDispatch

- 拆卡约束：
  - 所有任务必须标注 `里程碑引用 = M5`。
  - 推荐波次：
    - Wave1：Contracts 兼容扩展卡。
    - Wave2：Scene 变换输出卡 + Render 消费应用卡（可并行）。
    - Wave3：App 编排配套卡 + QA 复验卡。
- 范围限制：
  - 允许：契约字段扩展、Scene 帧产出、Render 提交逻辑、测试与门禁补齐。
  - 禁止：跨模块顺手重构、材质/资源系统扩张。
- 关键门禁：
  - 必须持续校验 `Engine.Render` 不直接引用 `Engine.Scene`。
  - 兼容路径（无显式 transform）必须通过回归验证。
