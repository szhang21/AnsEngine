# Plan Archive: PLAN-M4-2026-04-11

## GoalSummary

- 本轮目标：将渲染输入从“Render 内部硬编码 demo”升级为“Scene 驱动渲染提交”，完成可管理对象到画面输出的闭环。
- 成功标准（可验证）：
  - 至少 1 个场景节点可驱动画面变化（显示/隐藏或基础参数变化可见）。
  - `Engine.App` 不承载具体绘制逻辑，维持编排职责。
  - Build/Test/Smoke/Perf 门禁通过并可归档。

## Milestones

- M4: Scene -> Render 提交链路落地（唯一里程碑）
  - 交付结果：
    - `Engine.Scene` 提供最小可渲染数据输出（可被渲染模块消费）。
    - `Engine.Render` 消费场景输出完成绘制，而非内部写死单三角形。
    - `Engine.App` 仅负责主循环阶段衔接（更新/提交/渲染）。
  - 完成定义与验收口径：
    - Build: `dotnet build -c Debug` 与 `dotnet build -c Release` 通过。
    - Test: `dotnet test` 通过；新增/调整最小链路测试通过。
    - Smoke: 可启动并持续渲染 30s+，关闭后退出码 `0`。
    - Perf: 相比 M3 无明显退化（帧时间无异常抖动）。

## PriorityOrder

- P3: 暂不启用

## Risks

- high: 场景与渲染契约定义过重，导致一次性改动过大、跨模块回归难定位。
  - 规避策略：先落“最小可提交数据结构”，禁止一步到位做完整材质/资源体系。
- medium: `Engine.App` 可能被动吸收桥接细节，出现边界回退。
  - 规避策略：桥接接口放在 `Scene/Render` 边界，`App` 仅做时序编排。
- low: 现有 M3 演示路径被替换后缺少回退手段。
  - 规避策略：保留最小 fallback 开关或兼容路径用于冒烟对比。

## PlanningDecisions

- 默认选择：本轮只推进 M4，不并行推进 M5。
- 架构取舍：优先“数据契约最小化”而非“功能完备化”。
- 演示取舍：优先证明“场景可驱动画面”，不扩展材质系统与资源导入复杂度。
- 质量取舍：每次子步骤必须保持可编译、可运行，避免大爆改后集中修复。

## HandoffToDispatch

- 拆卡约束：
  - 所有任务必须标注 `里程碑引用 = M4`。
  - 主模块限定在 `Engine.Scene`、`Engine.Render`、`Engine.App`（其中 `Engine.App` 仅配套卡）。
- 范围限制：
  - 允许：最小场景渲染数据、提交流程、渲染消费、必要测试与门禁补齐。
  - 禁止：资源系统重构、材质系统扩展、跨模块顺手重构。
- 并行倾向：
  - 第一波单卡：先定最小 Scene/Render 契约（高风险，单独波次）。
  - 第二波并行：`Scene` 数据输出卡 + `Render` 消费卡。
  - 第三波收敛：`App` 编排配套 + QA 证据卡。
- 依赖建议：
  - 第二波任务依赖“契约卡完成”。
  - QA 卡必须依赖所有实现卡完成后执行。
