# Plan Archive: PLAN-M9-2026-04-22

## GoalSummary

- 本轮目标：把当前“meshId 命中 Render 内置顶点字典”的最小链路，升级为“磁盘 mesh 资产 -> Asset CPU 资产 -> Scene 引用 -> Render GPU 上传与绘制”的真实资产路径，并把模块边界收敛到可持续扩展的成熟引擎方向。
- 成功标准（可验证）：
  - 至少一个场景对象的 mesh 几何数据真实来自磁盘文件，而不是 `Engine.Render` 内置常量。
  - `Engine.Asset` 负责磁盘读取、格式导入、CPU 侧 mesh 资产构建与缓存；`Engine.Render` 不再保存内置 mesh 顶点表作为生产主路径。
  - `Engine.Scene` 继续只持有稳定资源引用，不持有导入器、文件路径解析或 GPU 资源细节。
  - `Engine.Render` 通过受控的 mesh 解析/缓存入口拿到规范化 mesh 数据，并负责 GPU buffer 生命周期；Asset 不直接创建 OpenGL 资源。
  - 设计上不把 OBJ 演示格式泄漏到公开主契约，后续可扩展到 glTF/引擎自有格式而不推翻边界。
  - `Scene / Asset / Render` 的引用关系保持稳定，不通过扩张 `Engine.Core` 来临时桥接业务接口。

## Risks

- high: 把 Asset 直接做成“返回 OpenGL-ready 顶点/缓冲句柄”的渲染前置模块，导致 CPU/GPU 资源边界混乱。
  - 规避策略：锁定 `Asset = 磁盘/导入/CPU 资产`，`Render = GPU 上传/缓存/释放`，禁止 Asset 直接依赖 OpenGL。
- high: 让 `Scene` 直接持有磁盘路径或导入器语义，后续场景存档、重定向、打包都会变脆弱。
  - 规避策略：Scene 只持有稳定 `meshId`/资产引用；磁盘位置由 Asset 目录或注册表解析。
- medium: 公开契约过早暴露特定导入格式，例如 OBJ 字段或第三方库返回类型，导致后续格式扩展困难。
  - 规避策略：对外只暴露引擎规范化 mesh 数据与 provider 接口，导入器实现细节留在 Asset 内部。
- medium: 把 provider/result/catalog 等业务桥接接口放进 `Engine.Core`，会让 Core 失去基础层定位，逐步变成跨模块杂糅层。
  - 规避策略：资源桥接接口统一放在 `Engine.Contracts`，Core 仅承载基础值类型和通用元信息。
- medium: 真实 mesh 接入后，Render 每帧重复做 CPU 展平或 GPU 上传，帧时间恶化。
  - 规避策略：明确“导入一次、缓存 CPU 数据、按需创建 GPU cache”的生命周期，并对共享 mesh 做缓存测试。
- low: 首个样例 mesh 过于简单，掩盖索引、法线或包围盒路径缺陷。
  - 规避策略：样例资源至少覆盖 indexed mesh，避免继续停留在纯 demo 三角形。

## PlanningDecisions

- 候选方向 A：`Asset` 直接返回渲染可用顶点数组，`Render` 只负责 draw。
  - 优点：短期改动少，容易出画面。
  - 缺点：会把渲染顶点布局、未来 GPU 资源策略和 Asset 强耦合，不符合成熟引擎职责分层。
- 候选方向 B（默认推荐）：`Asset` 输出引擎规范化 CPU mesh 资产，`Scene` 只传引用，`Render` 自行做 GPU 资源创建与缓存。
  - 优点：边界稳定，便于后续支持多格式导入、离线处理、热重载和多后端渲染。
  - 缺点：前期会多出一层数据模型与 provider 抽象。
- 默认推荐：选择方向 B。

- 候选桥接层 A（默认推荐）：继续扩展 `Engine.Contracts`，承载 `SceneMeshRef`、`MeshAssetData`、provider/result 等跨模块资源契约。
  - 优点：与现有 `Scene -> Contracts <- Render` 解耦方向一致，不污染 `Engine.Core` 的基础层职责。
  - 缺点：`Engine.Contracts` 会继续增长，需要保持只放稳定接口与只读数据结构。
- 候选桥接层 B：新增独立 `Engine.Asset.Contracts`。
  - 优点：资源契约更纯、更窄。
  - 缺点：当前阶段模块拆分偏早，会增加装配和维护成本。
- 候选桥接层 C：扩展 `Engine.Core` 作为资源桥接层。
  - 优点：模块数量不增加。
  - 缺点：会把 Core 从基础层推成业务中介层，长期最不稳定。
- 默认推荐：选择桥接层 A，不新增桥接模块，也不扩张 `Engine.Core`。

- 关键取舍 1：`meshId` 不直接等于文件路径。
  - 默认选择：引入磁盘清单文件 catalog，把稳定资源 ID 映射到磁盘源。
- 关键取舍 2：导入格式只是实现细节。
  - 默认选择：第一版先支持 `OBJ` 作为真实磁盘格式，但不把该格式暴露进公开主接口。
- 关键取舍 3：回退策略保留，但必须降级为错误处理路径。
  - 默认选择：仅在缺失/损坏资源时由 `Engine.Render` 回退到 fallback mesh，并记录可诊断信息。
- 关键取舍 4：本轮先做同步加载。
  - 默认选择：同步加载 + 缓存，接口保留未来异步/热重载扩展位，但不在本轮实现。
- 关键取舍 5：provider 获取方式。
  - 默认选择：由 `Engine.App` 把 `IMeshAssetProvider` 注入 `Engine.Render`，而不是让 Render 直接依赖 Asset 实现或全局定位器。

## HandoffToDispatch

- 拆卡约束：
  - 所有任务必须标注 `计划引用 = PLAN-M9-2026-04-22`。
  - 里程碑引用建议使用：
    - `M9.1`：边界/契约/数据模型
    - `M9.2`：磁盘导入与 Asset 缓存
    - `M9.3`：Scene/Render 真实资产接入
    - `M9.4`：扩展点与回归收口
- 推荐波次：
  - Wave1：Asset/Contracts 边界与 mesh 资产模型卡
  - Wave2：磁盘导入器与资产目录/缓存卡
  - Wave3：Render mesh provider 接入与 GPU cache 卡
  - Wave4：Scene 引用收敛与 App 装配卡
  - Wave5：QA 与门禁复验卡
- 主模块边界：
  - `Engine.Asset`：磁盘读取、导入、规范化 mesh 数据、缓存、句柄/记录、错误诊断
  - `Engine.Scene`：场景对象与资源引用关系，不做磁盘 IO、不做导入、不做 GPU 资源创建
  - `Engine.Render`：消费 mesh 引用，解析 CPU mesh，创建/缓存/释放 GPU 资源，执行绘制
  - `Engine.Contracts`：承载稳定资源引用和必要的跨模块查询接口，不承载导入器实现和 OpenGL 类型
  - `Engine.Core`：仅承载基础值类型与运行时元信息，不承载 mesh provider/catalog/result 等业务桥接职责
  - `Engine.App`：组合根装配 provider/service，不承载导入逻辑
- 依赖方向硬约束：
  - 允许：`Engine.Scene -> Engine.Contracts`
  - 允许：`Engine.Render -> Engine.Contracts`
  - 允许：`Engine.Asset -> Engine.Contracts`
  - 允许：`Engine.App -> Engine.Scene + Engine.Render + Engine.Asset + Engine.Contracts`
  - 允许：`Engine.Contracts -> Engine.Core`
  - 禁止：`Engine.Scene -> Engine.Asset`
  - 禁止：`Engine.Scene -> Engine.Render`
  - 禁止：`Engine.Render -> Engine.Scene`
  - 禁止：`Engine.Asset -> Engine.Render`
  - 禁止：`Engine.Asset -> Engine.Scene`
- 范围限制：
  - 允许：mesh 规范数据模型、磁盘导入、Asset 缓存、Scene 资源引用收敛、Render GPU cache、样例真实 mesh、测试和边界文档更新
  - 禁止：纹理系统、材质图、动画、ECS 大重构、场景编辑器、资源打包器
- 关键门禁：
  - 不允许 `Scene` 直接依赖导入器或磁盘路径解析
  - 不允许 `Asset` 直接创建或持有 OpenGL 资源
  - 不允许把资源桥接接口迁入 `Engine.Core`
  - 不允许 `Render` 继续以内置 mesh 字典作为生产主路径
  - 必须验证共享 mesh 的 CPU 缓存和 GPU 缓存命中
