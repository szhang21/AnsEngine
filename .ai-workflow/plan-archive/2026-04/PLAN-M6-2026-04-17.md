# Plan Archive: PLAN-M6-2026-04-17

## GoalSummary

- 本轮目标：将当前“CPU 直接改最终顶点”的 demo 级渲染路径，升级为由 `Model -> View -> Projection` 矩阵驱动的最小真实渲染管线。
- 成功标准（可验证）：
  - `Engine.Render` 不再依赖 CPU 预先算好的最终顶点位置，而是消费局部顶点数据与 MVP 矩阵输入。
  - 引入最小相机/视图契约，视图参数变化能真实影响画面结果。
  - 单对象渲染链路继续可见、可退出、不回退。
  - 保持依赖方向 `Scene -> Contracts -> Render`，`Render` 不直接引用 `Scene`。

## Milestones

- M6: 最小真实 MVP 渲染管线（唯一里程碑）
  - 交付结果：
    - 在 `Engine.Contracts` 中补齐最小相机/视图/投影相关契约，使 Scene 可输出对象变换与视图语义。
    - `Engine.Scene` 输出“对象 transform + 最小 camera/view 数据”，而不是继续仅靠 demo 顶点效果表达运动。
    - `Engine.Render` 改为使用 shader uniform 执行 MVP 矩阵变换，CPU 不再直接写最终裁剪空间顶点。
    - 保留最小 demo mesh，但它必须通过统一 mesh 数据入口参与提交，而不是散落在提交流程里。
  - 验收口径：
    - Build: `dotnet build -c Debug` / `dotnet build -c Release` 通过。
    - Test: `dotnet test` 通过；新增 MVP/相机链路测试通过。
    - Smoke: 启动后可见矩阵驱动结果，修改相机或模型变换时画面按预期变化，并可稳定退出。
    - Perf: 相比 M5 无明显帧时间退化。

## PriorityOrder

- P3: 暂不启用

## Risks

- high: 一次性引入相机、矩阵、shader uniform 后，容易把“可见输出”完全打坏。
  - 规避策略：先保留现有单对象三角形样例，只替换变换路径，不同时扩展材质和纹理。
- medium: 相机语义定义不清，导致 Contracts 过重或后续频繁改动。
  - 规避策略：本轮只定义最小 camera/view/projection 输入，不提前设计完整场景系统。
- medium: CPU 与 shader 侧坐标语义不一致，造成调试困难。
  - 规避策略：明确空间分层（Model/World/View/Clip），并补充 deterministic 测试。
- low: demo mesh 入口仍然半硬编码，导致后续资源接入改动面扩大。
  - 规避策略：本轮至少把 mesh 数据收敛到统一入口，为后续真实 mesh 资源做准备。

## PlanningDecisions

- 默认策略：先做“真矩阵管线”，暂不引入纹理系统、复杂材质系统和完整资源管线。
- 架构边界：Scene 提供对象与相机语义；Contracts 传输统一输入；Render 只负责消费与绘制；App 只做装配与编排。
- 实现取舍：优先保证空间语义正确，其次才是渲染能力扩展。
- 兼容策略：允许保留最小 triangle mesh 作为演示输入，但必须通过统一 mesh 数据入口进入 Render。
- 收敛规则：同模块内的顺序性增强默认并入主卡，不再为局部实现或局部收口单独新拆卡。

  - 禁止：纹理加载、复杂材质系统、完整资源导入管线、非目标跨模块重构。

  - 必须保留“可启动、可见、可退出”路径。
