# Plan Archive: PLAN-M7-2026-04-18

## GoalSummary

- 本轮目标：将当前仍偏 demo 的 mesh/material 占位语义升级为“最小真实资源输入”，让 `meshId` 和 `materialId` 真正参与渲染流程。
- 成功标准（可验证）：
  - `meshId` 能映射到真实 mesh 数据，而不是继续依赖内建固定三角形顶点。
  - `materialId` 能驱动最小材质参数，而不是仅通过 hash 派生颜色。
  - `Scene` 输出的资源引用可被 `Render` 解析并消费，且依赖方向仍保持 `Scene -> Contracts -> Render`。
  - 单对象与最小多对象路径均可稳定运行，不回退 `M6` 的真实 MVP 管线。

## Milestones

- M7: 最小真实资源输入（唯一里程碑）
  - 交付结果：
    - 建立最小 mesh 数据入口，使 Render 从统一 mesh 数据源读取局部顶点。
    - 建立最小 material 参数入口，使 Render 根据 `materialId` 获取基础渲染参数。
    - `Scene` 输出“对象 + transform + meshId + materialId”，Render 不再把资源语义写死在内部逻辑。
    - 保留最小 demo 资源集，但它们必须通过统一资源入口进入渲染链路。
  - 验收口径：
    - Build: `dotnet build -c Debug` / `dotnet build -c Release` 通过。
    - Test: `dotnet test` 通过；新增 mesh/material 解析与回退路径测试通过。
    - Smoke: 至少一个对象通过真实 `meshId` + `materialId` 成功渲染，运行稳定且可退出。
    - Perf: 相比 M6 无明显帧时间退化。

## PriorityOrder

- P0: M7（真实 mesh/material 输入落地）
- P1: 资源引用回退与最小多对象验证
- P2: 暂不启用
- P3: 暂不启用

## Risks

- high: 把资源解析、渲染表现、材质语义一次性绑太死，导致后续扩展受限。
  - 规避策略：本轮只做“最小真实资源入口”，不引入完整纹理系统或复杂材质体系。
- medium: 资源缺失/非法引用路径处理不清，导致运行时黑屏或崩溃。
  - 规避策略：为 mesh/material 缺失设计明确回退策略，并补测试覆盖。
- medium: Render 为了接真实资源又回退到内部写死逻辑。
  - 规避策略：要求 demo 资源也必须通过统一入口进入，不允许旁路。
- low: 多对象验证不足，导致资源入口只对单对象有效。
  - 规避策略：将“最小多对象渲染”纳入 smoke/回归范围。

## PlanningDecisions

- 默认策略：先做“最小真实资源输入”，不同时展开纹理加载、复杂 shader 变体和完整资源管线。
- 架构边界：Scene 只引用资源标识；Contracts 传输资源引用；Render 负责解析并提交；App 仅装配。
- 实现取舍：优先让 `meshId/materialId` 从“占位字段”变成“真实生效字段”，再考虑更复杂的资产来源。
- 兼容策略：允许保留内置 demo 资源，但必须通过统一资源解析入口访问。

## HandoffToDispatch

- 拆卡约束：
  - 所有任务必须标注 `里程碑引用 = M7`。
  - 推荐波次：
    - Wave1：Contracts 资源输入契约收敛卡。
    - Wave2：Render mesh 数据入口卡 + material 参数入口卡。
    - Wave3：Scene 资源引用输出卡。
    - Wave4：最小多对象与回退路径验证卡。
    - Wave5：QA 复验卡。
- 范围限制：
  - 允许：mesh/material 最小数据结构、解析入口、Scene 资源引用输出、测试与门禁补齐。
  - 禁止：完整纹理系统、复杂材质系统、磁盘资源导入管线、非目标跨模块重构。
- 关键门禁：
  - 必须验证 `meshId` 不再是空语义字段。
  - 必须验证 `materialId` 对最终渲染结果有真实影响。
  - 必须保持 `Render` 不直接依赖 `Scene`。
