# Plan Archive: PLAN-M4B-2026-04-13

## GoalSummary

- 本轮目标：将当前 `Engine.Render -> Engine.Scene` 的直接依赖演进为“Render 仅依赖渲染输入契约”，完成长期可维护的依赖反转。
- 成功标准（可验证）：
  - `Engine.Render` 不再直接引用 `Engine.Scene`，仅依赖契约层。
  - `Engine.Scene` 通过契约层提供渲染输入，`Engine.App` 完成装配注入。
  - 新增门禁：`Engine.Render.csproj` 禁止直接引用 `Engine.Scene.csproj`。
  - Build/Test/Smoke/Perf 门禁通过，并更新边界合同与归档证据。

## Milestones

- M4b: 契约下沉 + 依赖反转（唯一里程碑）
  - 交付结果：
    - 先更新 `.ai-workflow/boundaries/engine-render.md`，反映当前真实依赖，消除文档漂移。
    - 新建独立契约层（`Engine.Render.Contracts` 或 `Engine.Contracts`）承载 `SceneRenderFrame/SceneRenderItem` 与 provider 抽象。
    - 依赖关系调整为：`Engine.Render -> Contracts`、`Engine.Scene -> Contracts`，移除 `Engine.Render -> Engine.Scene`。
    - `Engine.App` 负责 provider 装配与注入，`Engine.Render` 仅消费契约。
    - 增加 csproj/CI 门禁检查，阻断 `Engine.Render` 回归引用 `Engine.Scene`。
  - 验收口径：
    - Build: `dotnet build -c Debug` 与 `dotnet build -c Release` 通过。
    - Test: `dotnet test` 通过，契约层与装配路径相关测试通过。
    - Smoke: 启动渲染行为与 M4 保持一致，可见输出不回退。
    - Perf: 相比 M4 无明显退化（帧时间与初始化时延无异常上升）。

## Risks

- high: 契约层拆分不当导致循环引用或编译依赖震荡。
  - 规避策略：先确定契约最小集合，再迁移实现，最后移除旧依赖。
- medium: 仅做 DI 不做契约下沉会留下编译期耦合。
  - 规避策略：强制接口定义迁出 `Engine.Scene`，并增加引用门禁。
- medium: 迁移阶段行为回归（渲染空帧、注入失败）。
  - 规避策略：保留 M4 冒烟基线，对比验证迁移前后可见结果一致。
- low: 边界文档未同步导致流程漂移。
  - 规避策略：将边界文档更新设为任务卡硬性验收项。

## PlanningDecisions

- 默认选择：采用“低风险渐进迁移”，顺序固定为“文档校准 -> 契约下沉 -> 依赖切换 -> 门禁加固”。
- 契约位置默认：优先 `Engine.Render.Contracts`；若后续跨子系统复用扩大，再评估收敛为 `Engine.Contracts`。
- 语义边界：`Render` 不认识 `Scene` 语义，只消费渲染输入契约。
- 质量策略：每步均可独立构建验证，不允许一次性大改后集中修复。

## HandoffToDispatch

- 拆卡约束：
  - 所有任务必须标注 `里程碑引用 = M4b`。
  - 推荐波次：
    - Wave1（单卡）：边界合同校准（`engine-render.md`）。
    - Wave2（并行）：契约层创建 + Scene 适配提供者。
    - Wave3（单卡）：Render 依赖切换与旧引用移除。
    - Wave4（并行）：App 装配注入 + 门禁规则落盘 + QA 证据卡。
- 范围限制：
  - 允许：契约类型、provider 抽象、项目引用调整、装配注入、门禁与边界文档更新。
  - 禁止：渲染功能扩展、资源系统重构、非目标跨模块重构。
- 并行倾向：
  - 高风险依赖切换卡独立波次执行。
  - QA/门禁卡后置并行，依赖实现卡全部完成。
- 关键门禁：
  - 必须新增“`Engine.Render.csproj` 不允许引用 `Engine.Scene.csproj`”校验。
  - 任何违反新依赖方向的变更不得进入 `Review`。
