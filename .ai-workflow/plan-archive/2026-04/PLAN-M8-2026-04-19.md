# Plan Archive: PLAN-M8-2026-04-19

## GoalSummary

- 本轮目标：将当前 Render 内部的内置 mesh 字典升级为“真实 Mesh 资产接入”，让 `meshId` 对应的几何数据来自明确资产来源，而不再是硬编码在渲染模块中的顶点数组。
- 成功标准（可验证）：
  - 至少一个对象通过真实 mesh 资产渲染，而不是内置三角形顶点数组。
  - mesh 数据来源从 `Engine.Render` 内部迁出，进入统一资产入口。
  - `Scene` 继续只引用 `meshId`，不关心 mesh 具体内容。
  - `Render` 保持只消费 Contracts/资产入口，不直接依赖 `Scene`。
  - 当前 MVP 管线继续成立，不因 mesh 来源替换而回退。

## Milestones

- M8: 真实 Mesh 资产接入（唯一里程碑）
  - 交付结果：
    - 建立最小 mesh 资产数据结构与读取入口，使 `meshId` 能映射到真实 mesh 数据。
    - 将 `Engine.Render` 中的内置三角形顶点字典迁移到统一 mesh 资产入口。
    - `Scene` 输出的 `meshId` 继续通过 Contracts 传递，Render 根据统一入口解析 mesh。
    - 保留最小 demo mesh 作为基础资产样例，但其来源必须是资产入口而非 Render 内部常量。
  - 验收口径：
    - Build: `dotnet build -c Debug` / `dotnet build -c Release` 通过。
    - Test: `dotnet test` 通过；新增 mesh 解析、缺失回退、最小多对象路径测试通过。
    - Smoke: 至少一个对象通过真实 mesh 资产成功渲染，运行稳定且可退出。
    - Perf: 相比 M7 无明显帧时间退化。

## PriorityOrder

- P0: M8（真实 mesh 资产接入）
- P1: mesh 缺失回退与多对象基础验证
- P2: 暂不启用
- P3: 暂不启用

## Risks

- high: mesh 资产入口设计过重，提前把完整资源系统复杂度引入当前阶段。
  - 规避策略：本轮只做最小 mesh 资产入口，不同时做纹理系统和完整导入管线。
- medium: Render 替换 mesh 来源后出现行为回退，导致现有 MVP 路径不稳定。
  - 规避策略：保留最小 demo mesh 作为资产样例，对比 M7 行为做 smoke 回归。
- medium: mesh 缺失或非法引用处理不清，造成运行时不可控失败。
  - 规避策略：定义明确回退策略，并增加缺失/非法引用测试。
- low: 资产入口只验证单对象，后续多对象时暴露隐藏问题。
  - 规避策略：把最小多对象路径纳入测试与 smoke。

## PlanningDecisions

- 默认策略：先做“真实 mesh 资产接入”，不同时引入完整纹理系统、复杂材质体系或磁盘资源导入管线。
- 架构边界：Scene 只持有 `meshId`；Contracts 负责资源标识传输；Asset/Render 共同完成 mesh 数据接入；App 仅负责装配。
- 实现取舍：先让 mesh 来源成真，再考虑纹理、材质扩展和更复杂的资产生命周期。
- 兼容策略：允许保留最小 demo mesh 资产，但它必须通过统一资产入口访问。

## HandoffToDispatch

- 拆卡约束：
  - 所有任务必须标注 `里程碑引用 = M8`。
  - 推荐波次：
    - Wave1：最小 mesh 资产契约/数据结构卡。
    - Wave2：Asset mesh 入口卡 + Render mesh 解析接入卡。
    - Wave3：Scene 资源引用适配卡。
    - Wave4：缺失回退与最小多对象验证卡。
    - Wave5：QA 复验卡。
- 范围限制：
  - 允许：mesh 资产最小数据结构、解析入口、Render 接入、Scene 资源引用适配、测试与门禁补齐。
  - 禁止：完整纹理系统、复杂材质系统、完整资源导入管线、非目标跨模块重构。
- 关键门禁：
  - 必须验证 `meshId` 真正驱动 mesh 数据来源。
  - 必须验证 demo mesh 也通过统一资产入口访问。
  - 必须保持 `Render` 不直接依赖 `Scene`。
