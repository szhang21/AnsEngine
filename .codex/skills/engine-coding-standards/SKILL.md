# 引擎编码规范（OpenTK + C#）

路径说明（硬规则）：
- `references/review-checklist.md` 指的是当前 skill 目录下文件：`.codex/skills/engine-coding-standards/references/review-checklist.md`。
- 该文件是评审规则来源（只读），不是项目归档产物，不要求出现在仓库根 `references/`。
- 任务执行与归档写入路径仍是 `.ai-workflow/**`。

使用本 Skill 统一代码改动风格，提升可维护性，并降低性能回退风险。

配套清单：
- 合并前请读取 `references/review-checklist.md`，并逐条给出通过/未通过结果。
- 模块边界定义请参考 `../engine-task-dispatch/references/boundary-contract-template.md`，先定边界再实现。
- 工程技术基线请参考 `../engine-task-dispatch/references/project-baseline.md`，实现与评审默认遵循该基线。

## 执行模式

所有非微小改动都按以下顺序执行：

1. 用一句话定义范围。
2. 标记受影响模块与边界约束。
3. 实施最小补丁，禁止跨模块顺手重构。
4. 执行构建/测试/性能检查。
5. 输出风险分级与后续建议。

不得跳过验证，不得静默扩大范围。
若任务与工程基线冲突，先走“基线变更任务卡”，不得直接修改实现。

## 1) 项目结构与命名

遵循以下规则：

- 每个文件仅保留一个主类型。
- 文件路径与命名空间保持一致。
- 类型、方法、属性使用 `PascalCase`。
- 局部变量与参数使用 `camelCase`。
- 接口统一 `I` 前缀。
- 命名优先体现领域语义（如 `RenderQueue`、`SceneNode`、`AssetHandle`）。
- 避免泛化命名（`Manager`、`Helper`、`Util`），除非职责确实通用且边界清晰。

建议采用以下分层：

- `Engine.Core`（日志、时间、基础抽象）
- `Engine.Platform`（窗口/输入/时钟桥接）
- `Engine.Render`（OpenGL 后端、Shader/材质/Mesh）
- `Engine.Scene`（实体/变换/层级/可见性）
- `Engine.Asset`（加载/缓存/重载）
- `Engine.PhysicsLite`（AABB/射线/基础碰撞）
- `Engine.App`（组合根、启动、主循环）

## 2) 模块边界规则

严格遵守边界：

- `Scene` 不得直接调用 OpenGL API。
- `Render` 不得持有玩法状态。
- `Platform` 不得承载渲染策略逻辑。
- `Asset` 不得直接修改场景图。
- `App` 作为组合根负责编排各模块。

新增 API 时必须明确：

- 模块负责什么
- 模块不负责什么
- 允许依赖哪些模块
- 对外最小公开接口是什么

除非用户明确要求架构迁移，否则拒绝任何违反边界的改动。

## 3) API 与类设计

API 保持收敛且显式：

- 优先使用构造注入传递依赖。
- 小接口优先于巨型接口。
- 可变状态默认私有，确需外部修改再放开。
- 返回清晰领域结果，避免隐式副作用。
- 对语义不直观的公开 API 补充简短 XML 注释。

避免以下模式：

- 全局静态可变状态
- 隐式跨模块回调
- “为了未来”而提前抽象的过度泛化设计

## 3.1) SOLID 原则（必遵守）

在引擎开发中，SOLID 不是口号，必须落实到可检查的设计决策。

- `S`（单一职责）：一个类只承担一个核心变化原因。  
  例：`MeshRenderer` 只负责渲染提交，不负责资源加载。
- `O`（开闭原则）：通过扩展支持新能力，避免修改稳定核心。  
  例：新增材质类型优先通过实现新策略/接口，而不是修改渲染主循环分支。
- `L`（里氏替换）：子类型必须可替换父类型而不破坏行为约定。  
  例：`IAssetLoader` 的实现必须遵守相同错误语义与生命周期约束。
- `I`（接口隔离）：接口保持小而专，避免“胖接口”。  
  例：将资源读取、反序列化、缓存管理拆成独立接口，而非 `IAssetSystem` 大一统。
- `D`（依赖倒置）：高层策略依赖抽象，底层细节实现抽象。  
  例：`Scene` 依赖 `IRenderer` 抽象，不直接依赖 OpenGL 调用细节。

违反 SOLID 的改动必须在评审中标记风险等级，并给出可执行的拆分或重构建议。

## 4) 异常处理与日志

统一错误策略：

- 不可恢复初始化失败使用异常。
- 可恢复分支使用返回结果（或可空值并显式判空）。
- 禁止静默吞异常。
- 日志级别统一：`Debug`、`Info`、`Warn`、`Error`。
- 日志必须包含上下文：子系统、资源 id/路径、操作名称。

热路径禁止逐帧噪声日志。

## 5) 性能规则（引擎向）

将帧循环视为高敏感热路径。

必须遵守：

- 在 Update/Render 循环中避免逐帧分配。
- 热路径避免 LINQ 与装箱。
- 可重复查询做缓存。
- 复用缓冲区与临时容器。
- 减少 OpenGL 状态切换与冗余绑定。
- 在当前范围内尽量进行批处理绘制。

引入额外开销时必须说明：

- 预期开销
- 可接受原因
- 回归衡量方式

## 6) 渲染专项规则（OpenTK/OpenGL）

- CPU 侧场景遍历与 GPU 命令提交分离。
- 资源生命周期显式管理（`Create`、`Use`、`Dispose`）。
- Shader 编译/链接失败输出可定位信息。
- GL 调用限制在渲染后端/适配层内部。
- 在开发构建中启用可接受开销的调试检查。

禁止把无关系统逻辑混入渲染后端。

## 7) 并发与生命周期

- 渲染上下文线程归属必须明确。
- 跨线程共享可变状态必须有明确同步策略。
- GPU 资源、窗口、服务按确定顺序关闭。
- `IDisposable` 所有权清晰，职责单一。

## 8) 测试与质量门禁

每次有效改动都必须满足：

1. `dotnet build -c Debug` 通过
2. `dotnet build -c Release` 通过
3. 受影响模块 `dotnet test` 通过
4. Demo 场景冒烟运行成功
5. 相比基线无明显帧时间退化

任一门禁失败，必须修复或给出明确阻塞说明。

## 9) 评审输出格式（必填）

进行评审/自评时，输出必须包含：

- `Scope`：一句话任务范围
- `BoundaryCheck`：通过/失败 + 违规项
- `CorrectnessRisks`：正确性风险（`high|medium|low`）
- `PerformanceRisks`：性能风险（分级）
- `MaintainabilityRisks`：可维护性风险（分级）
- `MustFixBeforeMerge`：合并前必须修复项
- `OptionalImprovements`：可选改进项
- `Validation`：构建/测试/运行/性能结果

结论必须具体、可执行。

## 10) AI 改动流程护栏

始终遵守：

- 优先最小、局部补丁
- 未被要求时保持公开 API 不变
- 不做范围外顺手重构
- 显式写出关键假设
- 与边界规则冲突时先澄清需求

未执行的验证不得声称已执行。

## 11) 完成定义

只有同时满足以下条件才算完成：

- 范围已实现
- 边界规则已满足
- 质量门禁通过（或阻塞已明确记录）
- 风险已列出并分级
- 已给出后续建议（如适用）
